# Spec-Driven Agentic Development

This project follows Spec-Driven Agentic Development. The knowledge base (`kb/`) is the source of truth. If kb/ is empty, you must build it first through ambiguity resolution.

## Methodology

### Phase 1: Ambiguity Resolution
Before writing any code, generate questions to eliminate all blind spots about the product. Iterate with the user until no ambiguities remain.

### Phase 2: Knowledge Base Creation
From the user's answers, create the KB. The KB is designed for **agent navigation** — agents need precise context cheaply (few tokens, few hops), not browsable prose.

**Core design principles:**
- Every file must be **self-sufficient** — an agent reading it in isolation must understand what it is, why it exists, and what to read next.
- Indexes are **routing tables**, not tables of contents — they answer "given my task, what do I load?"
- Prefer **many small files** over few large ones — agents can selectively load; they can't selectively ignore.
- **Encode intent in filenames** — agents use file trees as a first-pass context map.
- Each file < 200 lines. If longer, split by topic.

**Required KB structure:**
```
kb/
├── INDEX.md                    ← Master entry point (ALWAYS loaded first)
├── GLOSSARY.md                 ← Canonical terms & abbreviations
├── plan.md                     ← Implementation plan (Phase 3)
│
├── indexes/
│   └── by-task.md              ← "I need to do X → read files A, B, C"
│
├── domain/
│   ├── INDEX.md
│   └── prd.md                  ← Product Requirements Document
│
├── spec/
│   ├── INDEX.md
│   ├── data-model.md           ← Entities, fields, types, constraints
│   ├── algorithms.md           ← Sync protocol, state machines, pseudocode
│   ├── api-contracts.md        ← Inputs, outputs, error codes per interface
│   ├── config-and-formats.md   ← Config schema, file formats with examples
│   └── error-taxonomy.md       ← Every error type, when it occurs, user message
│
├── properties/
│   ├── INDEX.md
│   ├── functional.md           ← P1, P2, ... — invariants
│   ├── non-functional.md       ← NF1, NF2, ... — perf, security, reliability
│   └── edge-cases.md           ← T1, T2, ... — boundary conditions
│
├── architecture/
│   ├── overview.md             ← Module structure, dependency graph, patterns
│   └── decisions/              ← ADRs (Architecture Decision Records)
│       ├── INDEX.md
│       └── adr-NNN-<topic>.md
│
├── external/                   ← MANDATORY: one file per third-party SDK/API
│   ├── INDEX.md
│   └── <system>-sdk.md         ← Actual runtime behavior, NOT just the API surface
│
├── conventions/
│   ├── code-style.md
│   ├── error-handling.md
│   └── testing-strategy.md
│
├── runbooks/
│   └── audit-checklist.md
│
└── reports/                    ← Generated by auditors during quality loops
```

**The Master INDEX.md** — the agent's starting point. Must contain:
- 2-sentence project summary
- "How to use this KB" section with reading order
- Quick-load bundles table: goal → ordered list of files to load
- File count and last-updated date

**The `indexes/by-task.md`** — the core navigation tool. For each task type (implement, audit, debug, test), list the exact files to load in order, with a "key questions this answers" line.

**External Dependency Research** (files in `kb/external/`, MANDATORY per SDK/API):
- For every external system, document its **actual runtime behavior** — not just the public API.
- Read the SDK source code or official docs to identify: lazy-loading patterns (do relation fields trigger extra API calls?), pagination semantics, rate-limit policies, batching capabilities, known gotchas.
- Document the **request cost model**: how many API calls does each operation make? A "fetch issues" call that lazy-loads 5 relations per issue is 1 + 5N calls — this must be explicit.
- Derive concrete constraints: max requests per operation, bulk-fetch-then-join patterns, fields to request eagerly vs. resolve locally.
- This research happens BEFORE implementation.

**ADRs** (files in `kb/architecture/decisions/`):
- One ADR per significant design choice. Format: Context → Decision → Consequences → What this means for implementers.
- ADRs encode **why**, not just what. Agents making implementation decisions need this.

The PRD answers "are we building the right product?" The spec answers "are we building the product right?"

### Phase 3: Project Planning
Create `kb/plan.md` — an incremental implementation plan. Each step names modules, relevant properties, and acceptance criteria.

### Phase 4: Implementation (Ralph Loop)
For each step in the plan: implement → compile → test → fix → repeat until green.

### Phase 5: Quality Audit
Audit against the quality standards below. Fix all findings.

### Phase 6: Final Validation
Run the full test suite, CLI smoke tests, and verify all properties from the KB are covered.

## MANDATORY Quality Standards

Every source file MUST follow these rules. Violations are bugs.

### Literate Programming

Code should read like a technical document. Someone unfamiliar with the codebase should understand the intent, design choices, and assumptions by reading the code and its comments.

- **File header**: Every file starts with a block comment:
  ```
  // Module: <name> — <one-line purpose>
  //
  // This module is responsible for <what it does and why it exists>.
  // It implements <which spec features / properties>.
  // Key design decisions: <DI pattern, error handling approach, etc.>
  ```

- **Function documentation**: Every public function has JSDoc with:
  - `@param` with type AND semantic meaning (not just "the config" but "the sync configuration — must have been validated")
  - `@returns` describing what the return value represents
  - `@throws` listing every error type this function can throw and when
  - `@invariant P<N>` referencing which property from kb/properties/ this function helps enforce
  - `@example` for non-trivial functions showing typical usage

- **Inline comments — the WHY and the WHAT-IS-NOT-OBVIOUS**:
  - Every conditional branch: why this case exists, what it handles
  - Every algorithm step: what this step accomplishes in the larger flow
  - Every implicit assumption: what the code takes for granted that isn't obvious
  - Every trade-off: why this approach over the alternative
  - Every magic value: why this number, this threshold, this default

- **Target: comment ratio >= 20%** (including JSDoc)

### Contracts & Correctness

- Preconditions: every function documents what its inputs MUST satisfy
- Postconditions: every function documents what it guarantees about its output
- All errors are typed (custom error hierarchy with `userMessage` for humans)
- Reference properties from `kb/properties/` in JSDoc: `@invariant P4`
- No `any` types. Use `unknown` and narrow with type guards.

### Architecture

- **Entry point MUST be `src/index.ts`** — this is where commander is set up and all commands are registered. The first line must be `#!/usr/bin/env node`. The `package.json` "build" script must include `chmod +x dist/index.js` after `tsc` (TypeScript doesn't set execute permissions).
- Dependency injection: pass dependencies as parameters (interfaces, not implementations)
- Functions: max 30 lines. Files: max 200 lines.
- Clear module boundaries: no circular imports, no reaching into other module internals

### TypeScript & Module System (build-critical)

These are non-negotiable. Getting any of these wrong means the project does not build or run.

- **ESM project**: `package.json` has `"type": "module"`. All imports MUST use `.js` extensions: `import { foo } from './bar.js'` (even though the source file is `bar.ts`). This is required by Node.js ESM + TypeScript `"module": "nodenext"`.
- **tsconfig uses `"module": "nodenext"`**: this means `moduleResolution: "nodenext"` — bare imports resolve to `node_modules`, relative imports MUST have `.js` extension.
- **ESM-compatible dependencies only**: `chalk` v5+ (not v4), `ora` v6+ (not v5), `commander` v12+ are ESM. If you see `ERR_REQUIRE_ESM` at runtime, a dependency is CommonJS — check its version.
- **After EVERY code change**: run `npx tsc --noEmit` to verify compilation. Do NOT proceed if there are TypeScript errors.
- **Verify the CLI actually runs**: after implementing commands, run `npx tsx src/index.ts --help` and confirm it prints help with all commands listed. Code that compiles but doesn't run is a build failure.
- **Build script**: `package.json` "build" must be `"tsc && chmod +x dist/index.js"` — tsc alone doesn't set execute permissions.

### Testing

Testing must be comprehensive, multi-layered, and traceable to the specification.

#### Test Levels
- **Unit tests**: for every source file. Test individual functions in isolation with mocked dependencies.
- **Integration tests**: test module interactions (e.g., sync engine + file system + mapper working together).
- **End-to-end tests**: test complete CLI commands with realistic scenarios (init → pull → modify → push → verify).
- **Property-based tests**: for critical invariants, use randomized inputs to test that properties hold across many cases (e.g., roundtrip: for any valid issue, pull then push produces no change).
- **Edge case tests**: every edge case from kb/properties/edge-cases.md must have a dedicated test.
- **Error path tests**: every error type must have a test verifying it's thrown with the correct userMessage.

#### Test-Spec Traceability
- Every test name starts with the property or spec reference it verifies: `"P4: frontmatter→Linear→frontmatter roundtrip preserves all fields"`
- Every property in kb/properties/ must be covered by at least 2 tests (one happy path, one edge case)
- Group tests by property in describe blocks: `describe("P2: No data loss")`
- Create a test matrix in kb/reports/ mapping each property to its tests

#### Integration Tests (critical)
- At least one integration test must verify the REAL Linear API works (not mocked)
- Test that init can connect and verify the workspace
- Test that pull can fetch at least one issue from Linear
- These tests use the real LINEAR_API_KEY from the environment
- Mocks passing does NOT guarantee the real API integration works
- **READ-ONLY**: integration tests must NEVER create, update, or delete anything in Linear. Only read operations (list teams, list issues, fetch issue details). Use pull and status, never push without --dry-run.

#### Test Quality
- Mock external APIs in UNIT tests — but have at least one INTEGRATION test with the real API
- Use typed mocks that match the real interface — the mock MUST implement the exact same interface as the real client
- Assertions must be specific: assert exact values, not just "does not throw"
- Target: coverage >= 80%
- Target: at least 3 tests per source file on average

## Product Behavior Requirements (non-negotiable)

These behaviors MUST work correctly when tested manually against a real Linear workspace.
They are verified by manual testing against a real Linear workspace.

1. **Config format**: use `.kb-sync.json` (JSON, not YAML). One format, no ambiguity.
2. **Init is simple**: `kb-sync init` creates the config file. No team selection required — by default, all teams are synced.
3. **Pull fetches ALL issues from ALL teams by default**: no label filtering, no sync markers, no team restriction. A fresh `init` followed by `pull` MUST produce markdown files if the workspace has issues. The `--team <name>` flag optionally restricts to a single team.
4. **Error messages must include the real cause**: never wrap errors with a generic message that hides the actual problem. "Cannot reach Linear API" is useless — show what actually failed (e.g., "getTeam returned undefined for team ID xyz").
5. **Build must produce an executable**: `npm run build` must include `chmod +x dist/index.js` after `tsc`.
6. **Shebang required**: `src/index.ts` must start with `#!/usr/bin/env node`.

## KB Document Quality Standards

### Every KB content file MUST follow this template:
```markdown
---
id: <unique-id>
domain: <topic>
last-updated: <YYYY-MM-DD>
related: [<id1>, <id2>]
---

# Title

## One-liner
[What this covers in one sentence]

## Scope
What IS covered here. What is NOT (with pointer to where it lives).

## Key concepts
- **Term**: [definition] — see also `GLOSSARY.md#term`

## [Content sections]
...

## Agent notes
> When working with this, also check: `<related-file>`
> Gotcha: [something non-obvious that catches implementers]

## Related files
- `<file>` — why it's relevant
```

The **Agent notes** and **Related files** sections are critical — they act like compiler warnings guiding the agent toward files it didn't know it needed.

### INDEX.md files — routing tables, not prose
Every directory with more than one file gets an `INDEX.md`. It must answer: "given what I need, which file do I open?" Use a table format:

| If you need...              | Read this file            |
|-----------------------------|---------------------------|
| Data model field types      | `data-model.md`           |
| Sync algorithm pseudocode   | `algorithms.md`           |

### Content quality per file type

**PRD** (`domain/prd.md`) — "Are we building the right product?"
- User stories: "As a [user], I want [action] so that [benefit]"
- Commands with realistic usage examples (input → what happens → output)
- Non-functional expectations. Out of scope. Stakeholder-readable.

**Spec files** (`spec/`) — "Are we building the product right?"
- Split into focused files: data-model, algorithms, api-contracts, config-and-formats, error-taxonomy
- Each file covers ONE concern with enough detail to implement without questions

**Properties** (`properties/`) — "What must always be true?"
- Split into: functional (P1...), non-functional (NF1...), edge-cases (T1...)
- Each property: ID, name, formal statement, violation example, WHY it matters, test strategy

**ADRs** (`architecture/decisions/`) — "Why did we choose this?"
- Format: `status` (proposed/accepted/deprecated/superseded-by), Context, Decision, Consequences, What this means for implementers

**External** (`external/`) — "How does the third-party system actually behave?"
- Runtime behavior, request cost model, rate limits, pagination, lazy-loading gotchas
- Architectural constraints derived from the above
